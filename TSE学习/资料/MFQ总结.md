# MFQ测试分析方法总结

## 一、MFQ概述

**MFQ**是测试分析和测试设计的核心方法，代表三个测试维度：

- **M（Modeling）**：单功能建模
- **F（Function Interaction）**：功能交互
- **Q（Quality Attribute）**：质量属性

## 二、MFQ测试流程

MFQ测试流程遵循以下步骤：

```
KYM → TCO → Modeling → Tcon → TC
```

### 1. KYM（Know Your Mission）- 了解测试任务

**目标**：充分了解需求，信息收集，识别风险，预防性测试

**核心要点**：
- 主动获取更多相关信息，并不断修改完善，持续改进
- 对被测试对象的认知、上下文理解、对任务本身的认知

**CIDTESTD框架**：
- **C** - Customer（用户）
- **I** - Information（信息）
- **D** - Developer（开发者）
- **T** - Test（测试）
- **E** - Environment（环境）
- **S** - Schedule（进度）
- **T** - Test Item（测试项）
- **D** - Deliverable（交付件）

**原则**：把握主干，尽早开展，预防性，更多更好的test ideas

### 2. TCO（Testing Coverage Outline）- 测试覆盖大纲

**目标**：
1. 提取有价值信息
2. 将有价值信息进行重组
3. 提取有价值信息，并将其进行结构化处理

**横向逻辑**：MFQ, Bugs, Questions（问题）, Risks（风险）
- **重点**：提出疑问很重要

**纵向逻辑**：归纳、总结、梳理

### 3. Modeling（建模）

**建模步骤**：
1. 确定测试对象
2. 确定单功能边界
3. 单功能拆分
4. 挑单功能建模
5. 确定处理边界
6. 选择model
7. 确定参数
8. 识别业务规则
9. 判断规则完整性

**建模结构**：
```
Given（预置条件） → When（执行某个动作/输入） → Then（发生了什么/关注被测系统）
```

## 三、PPDCS建模方法

根据M的属性，选择合适的建模方法：

### 1. P - Process（流程）

**使用场景**：
- 需求中有明显的"业务流程"的含义
- 有多个步骤，各个步骤之间有一定的前后约束关系
- 所有步骤共同完成一件事情
- 整个过程可能涉及多于一个的执行者和触发者

**识别要点**：
- 识别操作
- 识别操作之间的逻辑关系
- 识别操作的流转的因素

**建模技术**：流程图技术

**适用工具**：流程图、判定表、判定树、因果图

### 2. P - Parameter（参数）

**使用场景**：
- 对象中存在很多参数，并且这些参数互相之间有一定的逻辑联系
- 没有明显的流程且涉及"很多参数"
- 设计规格中包含很多规则，每条规则有很多不同的变量和不同的值组成
- 参数的数量有限，能识别参数之间的逻辑关系

**识别要点**：
- 提炼出相关参数
- 理清逻辑关系
- 决定输入输出

**建模技术**：决策表、决策树

**适用工具**：判定表、判定树、因果图

### 3. D - Data（数据）

**使用场景**：
- 每个数据都有它特殊的范围值
- 不同数据的范围可能存在限制
- 涉及的数据个数是有限的
- 数据之间没有明显的规则或者逻辑关系

**建模技术**：等价类划分、边界值

### 4. C - Combination（组合）

**使用场景**：
- 因子个数多
- 每个因子存在多个可能的状态（水平）
- 因子之间可能存在一些逻辑约束关系

**识别要点**：
- 提炼相关因子
- 理清逻辑关系
- 输出因子的值

**建模技术**：正交试验、正交分析、正交矩阵、pairwise

**适用工具**：因子-状态表

### 5. S - State（状态）

**使用场景**：
- 涉及到多种状态，最好是针对同一个对象的多个状态
- 各个状态之间可以通过由于某种事件的发生互相转换

**三要素**：
- **状态**：多种状态
- **事件**：触发测试对象的事件
- **转移迁移**：状态之间的转换

**建模技术**：状态流转图

**与流程（P）的区别**：
- **P（流程）**：自上而下，流程基本不可逆
- **S（状态）**：无限制，状态可以来回转换

## 四、建模方法选择原则

**TEST原则**：
- **T** - Triggers（触发关键词）：抓住核心功能
- **E** - Essentials（关键要素）：尝试不同特征
- **S** - Spanning Differences（跨越差异）：围绕既定目标
- **T** - Targets（目标）

## 五、测试用例设计

**测试用例要素**：
- 测试输入
- 执行条件
- 预期结果

**测试用例原则**：
- **正确性**：用例设计正确
- **完备性**：覆盖全面
- **连贯性**：逻辑连贯
- **可判定性**：结果可判定
- **可执行性**：可实际执行

## 六、测试目标

测试的三大目标：
1. **证实**：验证功能正确性
2. **证伪**：发现缺陷
3. **预防缺陷**：通过测试设计预防问题

## 七、MFQ与需求测试流程

**需求测试流程**：
```
方案评审 → 测试策略、用例编写 → 需求评审 → 输出文本用例 → 测试执行
```

**MFQ流程**：
```
KYM → TCO → Modeling → Tcon → TC
```

## 八、实践要点

1. **KYM是对被分析对象上下文信息广度的思考、梳理、结构化的呈现**
   - 流程：收集信息 → 结构化梳理/呈现 → 识别关键点&风险 → 对TCO&测试策略以指引
   - 框架：who+why; what; how

2. **横向逻辑与纵向逻辑结合**
   - 横向：MFQ, Bugs, Questions, Risks
   - 纵向：归纳、总结、梳理

3. **建模方法的选择要根据需求特点**
   - 有明显流程 → 流程图
   - 多参数规则 → 决策表/决策树
   - 数据范围 → 等价类/边界值
   - 多因子组合 → 正交试验
   - 状态转换 → 状态图

## 九、总结

MFQ是一种系统化的测试分析和设计方法，通过KYM了解任务、TCO梳理覆盖、Modeling进行建模，最终输出高质量的测试用例。核心在于根据被测对象的特点选择合适的建模方法（PPDCS），确保测试的全面性和有效性。
