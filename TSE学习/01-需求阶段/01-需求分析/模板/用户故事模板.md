# 用户故事模板

## 标准格式

```
作为一个 [用户角色]
我想要 [功能描述]
以便于 [业务价值]
```

## 用户故事编写指南

### ✅ 好的用户故事特征（INVEST原则）

1. **Independent（独立的）**
   - 用户故事之间应尽可能独立，可以按任意顺序开发

2. **Negotiable（可协商的）**
   - 细节可以在开发过程中协商确定

3. **Valuable（有价值的）**
   - 对用户或业务有价值

4. **Estimable（可估算的）**
   - 开发团队能够估算工作量

5. **Small（小的）**
   - 可以在一个迭代内完成

6. **Testable（可测试的）**
   - 有明确的验收标准，可以验证完成

### 📝 编写技巧

1. **角色要具体**
   - ❌ 差：作为一个用户
   - ✅ 好：作为一个注册用户 / 作为一个管理员

2. **价值要明确**
   - ❌ 差：以便于更好的体验
   - ✅ 好：以便于我能够快速找到想要的产品

3. **验收标准要可测试**
   - ❌ 差：界面要好看
   - ✅ 好：当用户登录成功后，显示欢迎消息"欢迎，[用户名]"

4. **故事要足够小**
   - 如果故事太大，应该拆分成多个小故事

### 🎯 验收标准（AC）编写技巧

使用Given-When-Then格式：

```gherkin
场景：用户成功登录
  给定 用户已注册且账号有效
  当 用户输入正确的用户名和密码
  并 点击"登录"按钮
  那么 系统显示欢迎页面
  并且 用户状态更新为"已登录"
```

### 📊 用户故事拆分示例

**大故事（需要拆分）**：
```
作为一个电商用户
我想要管理我的购物车
以便于我可以方便地购买商品
```

**拆分后的小故事**：
1. 作为一个电商用户，我想添加商品到购物车，以便于我可以在结账时购买
2. 作为一个电商用户，我想从购物车移除商品，以便于我可以取消不需要的商品
3. 作为一个电商用户，我想修改购物车中商品的数量，以便于我可以调整购买数量
