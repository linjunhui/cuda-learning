# 验收标准（Acceptance Criteria）模板

## 什么是验收标准

验收标准是明确、可测试的条件，用于判断一个功能是否满足需求。好的验收标准应该：
- **明确**：清楚说明期望的行为
- **可测试**：可以通过测试验证
- **完整**：覆盖正常、异常、边界情况
- **简洁**：易于理解和执行

## 标准格式

### 格式1：Given-When-Then（推荐）

**场景**：[场景名称]

**给定（Given）**：
- [前提条件1]
- [前提条件2]

**当（When）**：
- [触发动作1]
- [触发动作2]

**那么（Then）**：
- [预期结果1]
- [预期结果2]
- [预期结果3]

**示例数据**：
| 输入1 | 输入2 | 预期输出 | 备注 |
|-------|-------|---------|------|
| 值A   | 值B   | 输出1   | 正常情况 |
| 值C   | 值D   | 输出2   | 边界情况 |

### 格式2：检查清单格式

**验收标准**：

功能完整性：
- [ ] 标准1：[具体描述]
- [ ] 标准2：[具体描述]
- [ ] 标准3：[具体描述]

性能要求：
- [ ] 标准1：[性能指标]
- [ ] 标准2：[性能指标]

安全要求：
- [ ] 标准1：[安全要求]
- [ ] 标准2：[安全要求]

## 模板示例

### 示例1：登录功能

**场景**：用户成功登录

**给定**：
- 用户已注册账号"testuser@example.com"
- 用户密码为"Password123"
- 账号状态为"激活"

**当**：
- 用户在登录页面输入邮箱"testuser@example.com"
- 用户输入密码"Password123"
- 用户点击"登录"按钮

**那么**：
- 系统验证用户名和密码正确
- 系统创建用户会话
- 系统跳转到主页
- 系统显示欢迎消息"欢迎，testuser"
- 系统记录登录日志

**异常场景1**：密码错误

**给定**：
- 用户已注册账号"testuser@example.com"
- 正确密码为"Password123"

**当**：
- 用户输入错误的密码"WrongPass"
- 用户点击"登录"按钮

**那么**：
- 系统验证失败
- 系统显示错误消息"用户名或密码错误"
- 系统不清除已输入的用户名
- 系统不创建用户会话

**边界场景1**：空密码

**给定**：
- 用户已注册账号"testuser@example.com"

**当**：
- 用户输入邮箱"testuser@example.com"
- 用户密码字段为空
- 用户点击"登录"按钮

**那么**：
- 系统显示验证错误"密码不能为空"
- 系统阻止表单提交

### 示例2：数据验证功能

**场景**：验证用户输入的数据格式

**给定**：
- 用户正在填写注册表单
- 表单包含邮箱字段和手机号字段

**测试用例**：

| 场景 | 输入值 | 预期结果 | 验证规则 |
|------|--------|---------|---------|
| 正常邮箱 | user@example.com | 通过验证 | 符合邮箱格式 |
| 无效邮箱1 | user@ | 验证失败，显示"邮箱格式不正确" | 缺少域名 |
| 无效邮箱2 | @example.com | 验证失败，显示"邮箱格式不正确" | 缺少用户名 |
| 正常手机号 | 13812345678 | 通过验证 | 11位数字，以1开头 |
| 无效手机号1 | 1381234567 | 验证失败，显示"手机号格式不正确" | 位数不足 |
| 无效手机号2 | 23812345678 | 验证失败，显示"手机号格式不正确" | 不是1开头 |

**那么**：
- 系统根据验证规则检查输入
- 验证通过时，允许提交表单
- 验证失败时，显示具体的错误消息
- 错误消息在用户修正输入后实时更新

## 验收标准编写指南

### ✅ 好的验收标准特征

1. **具体明确**
   - ❌ 差：界面要友好
   - ✅ 好：错误消息使用红色文字显示，位于输入框下方

2. **可测试**
   - ❌ 差：性能要好
   - ✅ 好：页面加载时间不超过2秒

3. **完整覆盖**
   - 包含正常流程
   - 包含异常流程
   - 包含边界情况

4. **用户导向**
   - 从用户角度描述
   - 关注业务价值

5. **简洁易懂**
   - 避免技术术语（除非必要）
   - 使用清晰的语言

### 📋 验收标准检查清单

在编写验收标准时，确保覆盖以下方面：

**功能完整性**：
- [ ] 正常流程是否完整描述？
- [ ] 异常流程是否考虑？
- [ ] 边界情况是否覆盖？
- [ ] 错误处理是否明确？

**数据验证**：
- [ ] 有效数据是否通过验证？
- [ ] 无效数据是否被拒绝？
- [ ] 边界值是否测试？
- [ ] 空值和null值是否处理？

**用户体验**：
- [ ] 界面反馈是否明确？
- [ ] 错误消息是否清晰？
- [ ] 操作流程是否顺畅？

**非功能需求**：
- [ ] 性能要求是否明确？
- [ ] 安全要求是否考虑？
- [ ] 兼容性是否覆盖？

### 🔄 与测试用例的关系

验收标准是测试用例的基础。每个验收标准应该对应至少一个测试用例：

```
验收标准 → 测试用例
场景：用户登录
  ↓
测试用例TC-001：验证用户使用正确密码登录成功
测试用例TC-002：验证用户使用错误密码登录失败
测试用例TC-003：验证空密码时显示错误提示
```

### 📝 验收标准示例库

**常见功能验收标准模式**：

1. **CRUD操作**：
   - 创建：验证必填字段、数据格式、创建成功
   - 读取：验证权限、数据完整性
   - 更新：验证可编辑字段、更新成功、版本控制
   - 删除：验证权限、确认提示、级联删除

2. **数据验证**：
   - 必填字段验证
   - 格式验证（邮箱、手机号、日期等）
   - 长度验证
   - 唯一性验证

3. **权限控制**：
   - 角色权限验证
   - 数据权限验证
   - 操作权限验证

## 使用说明

1. 为每个用户故事或功能需求编写验收标准
2. 在需求评审时，与团队一起审查验收标准
3. 将验收标准作为开发指南和测试依据
4. 在开发过程中，如有疑问以验收标准为准
5. 功能完成后，使用验收标准进行验收测试
