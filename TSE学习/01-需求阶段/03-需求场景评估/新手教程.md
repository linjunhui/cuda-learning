# 需求场景评估 - 新手教程

## 📖 什么是需求场景评估？

需求场景评估就是：**检查需求描述的场景是否正确、是否完整**。

### 简单理解

**场景** = 用户使用系统的具体情况

**例子**：
```
需求：用户登录功能

场景1：用户用正确密码登录 ✅
场景2：用户用错误密码登录 ✅
场景3：用户忘记密码怎么办？❓（需求文档没提到）
场景4：用户账号被锁定了怎么办？❓（需求文档没提到）

需求场景评估：
  - 场景1和2是正确的
  - 场景3和4缺失了，需要补充
```

### 为什么要评估需求场景？

**没有场景评估**：
```
需求：用户登录功能
上线后用户："我忘记密码了，怎么办？"
开发："啊，需求文档没写这个场景..."
→ 功能不完整，需要补开发
```

**有场景评估**：
```
需求：用户登录功能
场景评估时发现："忘记密码的场景缺失"
补充：添加忘记密码功能
→ 需求完整，功能完善
```

## 第一部分：场景正确性验证（1级）

### 1.1 什么是场景正确性？

**场景正确性** = 需求描述的场景是否符合实际情况

**验证方法**：
1. 场景是否符合用户使用习惯？
2. 场景是否符合业务规则？
3. 场景是否可以实现？

**例子1：正确的场景**

**需求描述**：
```
场景：用户输入用户名和密码，点击登录
```

**验证**：
- ✅ 符合用户使用习惯：用户确实会这样做
- ✅ 符合业务规则：登录流程正确
- ✅ 可以实现：技术上可行

**结论**：场景正确 ✅

**例子2：不正确的场景**

**需求描述**：
```
场景：用户输入用户名后，系统自动填充密码并登录
```

**验证**：
- ❌ 不符合安全规则：自动填充密码不安全
- ❌ 不符合用户习惯：用户不希望系统自动登录
- ❌ 安全风险：密码泄露风险

**结论**：场景不正确，需要修改 ❌

### 1.2 如何验证场景正确性？

**检查清单**：

- [ ] **场景是否符合实际情况？**
  - 用户真的会这样使用吗？
  - 实际情况是这样吗？

- [ ] **场景是否符合业务规则？**
  - 是否符合公司的业务规则？
  - 是否符合行业规范？

- [ ] **场景是否符合安全要求？**
  - 是否存在安全风险？
  - 是否满足安全标准？

- [ ] **场景是否可以实现？**
  - 技术上是否可行？
  - 资源是否充足？

**例子**：验证购物车场景

**需求描述的场景**：
```
场景1：用户添加商品到购物车
场景2：用户从购物车删除商品
场景3：用户修改商品数量
```

**验证结果**：
- ✅ 场景1正确：用户确实会添加商品
- ✅ 场景2正确：用户确实会删除商品
- ✅ 场景3正确：用户确实会修改数量

**但是发现缺失**：
- ❓ 场景4：购物车为空时用户点击结算 → 缺失
- ❓ 场景5：商品库存不足时添加到购物车 → 缺失

## 第二部分：场景完备性验证（2级）

### 2.1 什么是场景完备性？

**场景完备性** = 需求是否包含了所有重要的使用场景

**完备性检查**：
1. 正常场景（主流程）✓
2. 异常场景（错误处理）？
3. 边界场景（特殊情况）？
4. 扩展场景（其他可能）？

### 2.2 如何检查场景完备性？

#### 方法1：按流程检查

**步骤**：
1. 画出主流程（正常场景）
2. 识别每个步骤可能的问题
3. 补充异常场景
4. 补充边界场景

**例子**：登录功能场景检查

**主流程（正常场景）**：
```
1. 用户打开登录页面 ✓
2. 用户输入用户名和密码 ✓
3. 用户点击登录 ✓
4. 系统验证成功 ✓
5. 用户登录成功 ✓
```

**异常场景识别**：
- 步骤2异常：输入为空、输入格式错误
- 步骤4异常：用户名不存在、密码错误
- 步骤5异常：账号被锁定、账号未激活

**边界场景识别**：
- 密码长度边界：最短密码、最长密码
- 用户名格式边界：最长邮箱、最短手机号

#### 方法2：按用户角色检查

**不同用户角色的场景**：
- 新用户：注册流程
- 老用户：登录流程
- 管理员：管理功能
- 访客：浏览功能

#### 方法3：按环境检查

**不同使用环境的场景**：
- 网络正常：正常流程
- 网络断开：离线处理
- 网络慢：超时处理
- 不同浏览器：兼容性

### 2.3 发现BA/SE未考虑到的场景

**BA = 业务分析师**
**SE = 系统工程师**

**你的任务**：发现他们遗漏的场景

**常见遗漏的场景**：

1. **错误处理场景**
   - BA/SE可能只考虑正常流程
   - 你需要考虑各种错误情况

2. **边界场景**
   - BA/SE可能只考虑典型情况
   - 你需要考虑边界情况

3. **用户体验场景**
   - BA/SE可能只考虑功能
   - 你需要考虑用户感受

4. **集成场景**
   - BA/SE可能只考虑单个功能
   - 你需要考虑与其他功能的交互

**例子**：发现遗漏的场景

**BA/SE考虑的场景**：
```
场景1：用户添加商品到购物车 ✓
场景2：用户修改商品数量 ✓
场景3：用户结算 ✓
```

**你发现的遗漏场景**：
```
场景4：购物车为空时用户点击结算
  → 应该提示"购物车为空"

场景5：商品在购物车中时，商品下架了
  → 应该提示"商品已下架，已从购物车移除"

场景6：商品价格变化了
  → 应该提示"商品价格已更新"

场景7：用户长时间未操作，购物车过期
  → 应该提示或清空购物车
```

**价值**：让需求更完整，避免上线后的问题 ✅

## 第三部分：实战教程

### 实战案例：评估"忘记密码"功能的场景

#### 第一步：理解需求

**需求描述**：
```
功能：用户忘记密码时可以重置密码
流程：
  1. 用户点击"忘记密码"
  2. 用户输入邮箱
  3. 系统发送重置链接
  4. 用户点击链接重置密码
```

#### 第二步：验证场景正确性（1级）

**场景1：用户输入正确的邮箱**
- ✅ 符合用户习惯
- ✅ 符合业务规则
- ✅ 可以实现

**场景2：用户输入错误的邮箱**
- ✅ 需要验证邮箱是否存在
- ✅ 需要给出错误提示
- ✅ 场景正确

**结论**：主要场景正确 ✅

#### 第三步：检查场景完备性（2级）

**正常场景**：
- ✅ 用户输入正确邮箱，收到邮件，重置成功

**异常场景（发现遗漏）**：
1. ❓ 用户输入不存在的邮箱 → 缺失
2. ❓ 用户输入邮箱格式错误 → 缺失
3. ❓ 重置链接过期了 → 缺失
4. ❓ 重置链接被使用过了 → 缺失

**边界场景（发现遗漏）**：
1. ❓ 用户多次点击"发送重置邮件" → 缺失
2. ❓ 重置链接在不同设备上使用 → 缺失

**补充后的场景**：

```
场景1：正常重置流程 ✓
场景2：邮箱不存在 → 提示"邮箱未注册"
场景3：邮箱格式错误 → 提示"邮箱格式不正确"
场景4：重置链接过期（24小时后） → 提示"链接已过期，请重新申请"
场景5：重置链接已使用 → 提示"链接已失效，请重新申请"
场景6：频繁发送邮件（1分钟内3次） → 限制发送频率
场景7：在不同设备使用链接 → 验证安全性
```

**结论**：补充了7个遗漏场景，需求更完整 ✅

## 常见问题

### Q1：如何判断场景是否重要？

**A**：
- 用户使用频率高 = 重要
- 影响核心功能 = 重要
- 容易出错 = 重要

### Q2：场景太多怎么办？

**A**：
- 按优先级排序
- 重要场景先实现
- 次要场景后续补充

### Q3：如何发现遗漏的场景？

**A**：
- 使用场景检查清单
- 参考类似功能的场景
- 咨询有经验的同事
- 在实际使用中发现
