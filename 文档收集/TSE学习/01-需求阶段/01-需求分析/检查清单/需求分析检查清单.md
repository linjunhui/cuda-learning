# 需求分析检查清单

## 使用说明

本检查清单用于评审需求分析的质量。在需求评审时，使用此清单确保需求分析的完整性和质量。

---

## 1. 需求完整性检查

### 1.1 基本信息
- [ ] 需求标题清晰明确
- [ ] 需求编号唯一标识
- [ ] 需求版本信息完整
- [ ] 创建人和日期记录
- [ ] 最后更新日期和人员记录

### 1.2 背景和上下文
- [ ] 业务背景描述清楚
- [ ] 问题陈述明确
- [ ] 需求来源清晰（用户、业务、合规等）
- [ ] 相关历史需求已关联

### 1.3 范围和边界
- [ ] 功能范围明确界定
- [ ] 明确说明包含的功能
- [ ] 明确说明不包含的功能
- [ ] 边界情况已考虑

---

## 2. 需求清晰性检查

### 2.1 需求描述
- [ ] 使用清晰、无歧义的语言
- [ ] 避免使用模糊词汇（如"好"、"快速"、"友好"）
- [ ] 技术术语已解释或定义
- [ ] 专业术语使用一致

### 2.2 具体性
- [ ] 需求描述具体而非抽象
- [ ] 使用具体例子而非抽象描述
- [ ] 量化指标明确（如性能指标）
- [ ] 验收标准可测试

---

## 3. 需求一致性检查

### 3.1 内部一致性
- [ ] 需求文档内部无矛盾
- [ ] 术语使用一致
- [ ] 格式规范统一
- [ ] 编号规则一致

### 3.2 外部一致性
- [ ] 与相关需求无冲突
- [ ] 与现有系统设计一致
- [ ] 与业务规则一致
- [ ] 与技术规范一致

---

## 4. 需求可追溯性检查

### 4.1 向上追溯
- [ ] 可以追溯到业务目标
- [ ] 可以追溯到用户需求
- [ ] 可以追溯到原始需求来源

### 4.2 向下追溯
- [ ] 可以分解为功能点
- [ ] 可以关联到设计文档
- [ ] 可以关联到测试用例
- [ ] 可以关联到实现代码

---

### 5.1 技术可行性
- [ ] 技术方案可行
- [ ] 技术约束已识别
- [ ] 技术风险已评估
- [ ] 技术依赖已明确

### 5.2 资源可行性
- [ ] 人力资源充足
- [ ] 时间资源合理
- [ ] 预算资源充足
- [ ] 外部资源可获得

### 5.3 业务可行性
- [ ] 业务规则支持
- [ ] 业务流程兼容
- [ ] 业务价值明确
- [ ] 业务影响已评估

---

## 6. 需求优先级检查

### 6.1 优先级定义

- [ ] 优先级评估依据明确

### 6.2 优先级合理性

### 7.4 兼容性需求

- [ ] 版本兼容性要求

## 综合评分

### 评分标准
- **优秀（90-100分）**：所有关键项都满足，需求质量很高
- **良好（75-89分）**：大部分项满足，有少量改进空间
- **一般（60-74分）**：基本项满足，需要较多改进
- **待改进（<60分）**：关键项缺失，需要重大改进
