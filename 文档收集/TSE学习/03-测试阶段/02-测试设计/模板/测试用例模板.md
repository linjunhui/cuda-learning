# 测试用例模板

## 标准测试用例格式

### 基本格式

```
测试用例ID：[TC-模块-序号]
测试用例名称：[简洁描述测试内容]
优先级：P0/P1/P2
测试类型：功能测试/性能测试/安全测试等
所属模块：[模块名称]
创建人：[姓名]
创建日期：YYYY-MM-DD
最后更新：YYYY-MM-DD
```

## 详细测试用例模板

### 模板1：功能测试用例（推荐）

```
测试用例ID：TC-LOGIN-001
测试用例名称：用户使用正确邮箱和密码登录成功
优先级：P0
测试类型：功能测试
所属模块：用户管理-登录功能
前置条件：
  1. 系统中存在测试用户：
     - 邮箱：test@example.com
     - 密码：Test123456
     - 账号状态：激活
  2. 用户未登录状态
  3. 浏览器可以正常访问系统

测试步骤：
  步骤1：访问登录页面
    操作：在浏览器地址栏输入登录URL
    预期：页面加载成功，显示登录表单

  步骤2：输入邮箱
    操作：在邮箱输入框输入"test@example.com"
    预期：邮箱输入框显示输入内容

  步骤3：输入密码
    操作：在密码输入框输入"Test123456"
    预期：密码输入框显示为密文（*或●）

  步骤4：点击登录按钮
    操作：点击"登录"按钮
    预期：系统开始验证，显示加载提示

预期结果：
  1. 系统验证用户名和密码匹配成功
  2. 系统创建用户会话（生成session/cookie）
  3. 页面显示"登录成功"提示消息（1-2秒后消失）
  4. 系统自动跳转到主页
  5. 主页显示欢迎消息"欢迎，test"
  6. 浏览器URL更新为主页地址
  7. 登录日志记录登录时间和IP地址

实际结果：（执行后填写）
  通过 / 失败 / 阻塞

失败原因：（如果失败，填写原因）

关联缺陷：（如果有缺陷，填写缺陷ID）
```

```
测试用例ID：TC-REG-001

优先级：P0
测试类型：功能测试-边界值测试
所属模块：用户管理-注册功能

前置条件：
  - 系统密码规则：6-20位，包含字母和数字
  - 用户访问注册页面

测试数据：
  密码：Test12（6位，包含字母和数字）

测试步骤：
  1. 用户填写其他必填信息（邮箱、手机号等）
  2. 用户在密码输入框输入"Test12"
  3. 用户在确认密码输入框输入"Test12"
  4. 用户点击"注册"按钮

预期结果：
  1. 系统验证密码长度=6，符合要求（≥6）
  2. 系统验证密码包含字母和数字，符合要求
  3. 系统验证两次密码输入一致
  4. 系统创建账号成功
  5. 系统显示"注册成功"消息

实际结果：
  通过 / 失败 / 阻塞

备注：
  - 验证下边界值（最小值）
  - 同时验证上边界值用例TC-REG-002（20位）
```

## 测试用例编写规范

### 1. 用例ID命名规范

**格式**：TC-模块缩写-序号

**示例**：
- TC-LOGIN-001（登录模块第1个用例）
- TC-REG-001（注册模块第1个用例）
- TC-CART-001（购物车模块第1个用例）

**模块缩写建议**：
- LOGIN：登录
- REG：注册
- CART：购物车
- PAY：支付
- ORDER：订单

### 2. 优先级划分

**P0（必须）**：
- 核心功能
- 主流程
- 影响用户基本使用的功能

**P1（重要）**：
- 重要功能
- 异常流程
- 影响用户体验的功能

**P2（一般）**：
- 次要功能
- 边界情况
- 优化类功能

### 3. 测试步骤编写原则

**原则1：清晰明确**
- ✅ 好："在邮箱输入框输入test@example.com"
- ❌ 差："输入邮箱"

**原则2：可执行**
- ✅ 好："点击页面右上角的'登录'按钮"
- ❌ 差："进入登录页面"

**原则3：独立性**
- 每个用例可以独立执行
- 不依赖其他用例的执行顺序

### 4. 预期结果编写原则

**原则1：具体可验证**
- ✅ 好："系统显示'登录成功'消息，字体为绿色，位于页面顶部"
- ❌ 差："系统显示成功"

**原则2：完整**
- 包含界面变化
- 包含数据变化
- 包含系统状态变化

**原则3：可测量**
- 使用具体数值（如响应时间<2秒）
- 使用可观察的状态（如按钮变为灰色）

## 测试用例管理

### 用例状态

- **待执行**：用例已编写，未执行
- **执行中**：正在执行用例
- **通过**：用例执行成功，结果符合预期
- **失败**：用例执行失败，实际结果不符合预期
- **阻塞**：因环境或依赖问题无法执行
- **跳过**：当前版本不执行此用例

### 用例维护

- **需求变更时**：更新相关用例
- **发现缺陷后**：补充用例覆盖缺陷场景
- **定期评审**：检查用例是否仍然有效
- **持续优化**：根据经验改进用例质量
